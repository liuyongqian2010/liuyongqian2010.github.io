jenkins自动化部署--部署war包到远程服务器

jenkins自动化部署–部署war包到远程服务器

1. Jenkins插件安装

  • 在系统管理—插件管理下，安装Publish over SSH插件

  • 在系统管理—系统设置下，配置SSH插件信息

    #只填写必要信息
SSH Server

    Name       testing-yuancheng（随意一个名字即可）
    Hostname   192.168.110.74（需要部署的远程服务器IP）
    Username   root（登录远程服务器的用户名）
    Remote Directory  /usr/tomcat/webapps/（远程服务器工作目录）
    √ Use password authentication, or use a different key
    Passphrase / Password  （输入远程服务器登录密码）
    Port           22
    Timeout (ms)   300000

2. 构建项目

  • 2.1 新建任务（输入一个任务名称，如testing001，选择“构建一个自由风格的软件项目”）
  • 2.2 源码管理（选择“Git”，输入仓库URL和git账号）
  • 2.3 构建
    • 调用顶层Maven目标,目标：clean package -Dmaven.test.skip=true

  • 2.4 构建后操作

    • 选择“Send build artifacts over SSH”

      Transfer Set

    Source files        target/*.war
    Remove prefix         target/
    Remote directory    #可不写，默认系统设置项/usr/tomcat/webapps/
    Exec command        sudo sh /usr/deploy_cmp.sh

    • deploy_cmp.sh，执行远程部署war包的命令，shell内容可以参考如下：

      #!/bin/bash
export BUILD_ID=dontkillme
TOMCATTHREAD=`ps aux | grep 'tomcat' | grep -v grep`
if test -z "$TOMCATTHREAD"
then 
    echo "TOMCAT NOT START"
else  
    THREADLIST=(${TOMCATTHREAD// / })
    PID=${THREADLIST[1]}   
    kill -9 $PID   
    sleep 5s
fi  
    cd /usr/tomcat/webapps/   
    rm -f cmp.war  
    rm -rf cmp 
    cd /usr/tomcat/bin
    ./startup.sh
    sleep 5s

        location /api {
            proxy_pass http://192.168.110.**:8080/cmp;  #后台接口
            }
        }

GCC升级

GCC升级

1. 选择升级版本

  • Linux系统默认安装的gcc版本均在4.9以下，一些软件比如NodeJS的编译需要比较高的版本
  • 选择需要升级的版本，http://mirror.koddos.net/gcc/releases/ ，当前选择版本为：gcc-7.4.0.tar.gz

2. 解压源码包

  • 将压缩包放在/usr/local下，进行解压：tar -zvxf gcc-7.4.0.tar.gz
  • 装备bzip2软件，之后的编译需要，执行：yum install bzip2

3. 开始编译安装

  • 下载供编译需要的依赖项

    cd /usr/local/gcc-7.4.0/ 后执行命令：
./contrib/download_prerequisites

  • 编译安装

    mkdir build  新建一个文件夹存放编译文件
cd build     后进行编译：
../configure --enable-languages=c,c++ --enable-checking=release  -disable-multilib  仅安装64位执行
make         生成Makefile文件，该过程可能会很耗时，起码90分钟
make install 之后执行安装命令

4. 解决运行程序gcc出现’GLIBCXX_3.4.21’ not found

  • 4.1 具体错误

    /usr/node-v10.15.3/out/Release/torque: /lib64/libstdc++.so.6: version `GLIBCXX_3.4.20' not found (required by /usr/node-v10.15.3/out/Release/torque)

  • 4.2 问题原因：升级gcc时，生成的动态库没有替换老版本 gcc 动态库导致

  • 4.3 解决方案：将gcc最新版本的动态库替换系统中老版本的动态库。

    • 查找编译gcc时生成的最新动态库 

      find / -name "libstdc++.so*"，
最新动态库为：/usr/local/gcc-7.4.0/build/stage1-x86_64-pc-linux-gnu/libstdc++-v3/src/.libs/libstdc++.so.6.0.24

    • 将找到的动态库libstdc++.so.6.0.24复制到/usr/lib64下

      cp /usr/local/gcc-7.4.0/build/stage1-x86_64-pc-linux-gnu/libstdc++-v3/src/.libs/libstdc++.so.6.0.24 /usr/lib64

    • 切换工作目录至/usr/lib64，删除原来的软连接，将默认库的软连接指向最新动态库。

      cd /usr/lib64
rm -rf libstdc++.so.6
ln -s libstdc++.so.6.0.24 libstdc++.so.6

5. 重启服务器，验证版本

  • 执行命令：gcc -v

MySQL数据库读写分离

MySQL数据库读写分离

1. 简介

  • 通过mysql-proxy实现读写分离之前需要lua脚本实现的，现在mysql-proxy里面已经集成了lua，无需另外安装lua。
  • 目前mysql-proxy的版本：mysql-proxy-0.8.5-linux-glibc2.3-x86-64bit。

2. 安装

  • 将安装包上传到/usr/local/目录下，并解压：tar -zvxf mysql-proxy-0.8.5-linux-glibc2.3-x86-64bit.tar.gz
  • 重新命名解压后的安装包：mv mysql-proxy-0.8.5-linux-glibc2.3-x86-64bit/ mysql-proxy

3. 配置mysql-proxy

  • 3.1 主配置文件

    cd /usr/local/mysql-proxy/
mkdir lua        #创建脚本存放目录
mkdir logs       #创建日志目录
cp share/doc/mysql-proxy/rw-splitting.lua ./lua     #复制读写分离配置文件
cp share/doc/mysql-proxy/admin-sql.lua ./lua        #复制管理脚本
vi /etc/mysql-proxy.cnf              #创建配置文件

[mysql-proxy]
user=root                            #运行mysql-proxy用户
admin-username=mysql                 #主从mysql数据库共有的用户
admin-password=123456                #用户的密码
proxy-address=192.168.110.72:4040    #mysql-proxy运行ip和端口，端口默认4040
proxy-read-only-backend-addresses=192.168.110.74:3306      #指定后端从slave数据库读取数据
proxy-backend-addresses=192.168.110.73:3306                       #指定后端从主master数据库写入数据
proxy-lua-script=/usr/local/mysql-proxy/lua/rw-splitting.lua     #指定读写分离配置文件位置
admin-lua-script=/usr/local/mysql-proxy/lua/admin-sql.lua      #指定管理脚本
log-file=/usr/local/mysql-proxy/logs/mysql-proxy.log              #日志位置
log-level=info                         #定义log日志级别，由高到低分别有(error|warning|info|message|debug)
daemon=true                            #以守护进程方式运行
keepalive=true                         #mysql-proxy崩溃时，尝试重启

保存退出后，
赋予配置文件权限：chmod 660 /etc/mysql-proxy.cnf

  • 3.2 读写分离配置文件

    vi /usr/local/mysql-proxy/lua/rw-splitting.lua

if not proxy.global.config.rwsplit then
 proxy.global.config.rwsplit = {
  min_idle_connections = 1,            #默认超过4个连接数时，才开始读写分离，改为1
  max_idle_connections = 1,            #默认8，改为1
  is_debug = false
 }
end

#注意：mysql-proxy会检测客户端连接，当连接没有超过min_idle_connections预设值时, 不会进行读写分离, 即查询操作会发生到Master上。

4. 启动mysql-proxy

  • 启动命令：/usr/local/mysql-proxy/bin/mysql-proxy --defaults-file=/etc/mysql-proxy.cnf
  • 查看命令：netstat -tupln | grep 4040 或者 netstat -antpl|grep mysql
  • 关闭命令：sudo kill 进程号

5. 打开防火墙的4040端口

  • 执行命令：/sbin/iptables -I INPUT -p tcp --dport 4040 -j ACCEPT

6. 测试读写分离

  • 6.1 在主数据库服务器创建用于mysql-proxy连接使用的用户密码：mysqlproxy 123456，从服务器也会同步这个操作。

    需要进入主数据库，执行：
grant all on *.* to 'mysqlproxy'@'192.168.110.72' identified by '123456';
刷新执行：
flush privileges;

  • 6.2 使用客户端连接mysql-proxy

    命令：mysql -h 192.168.110.72 -P 4040 -u mysqlproxy -p123456

创建数据库和表，这时的数据只写入主mysql，然后再同步从slave。
mysql> create table work_user (number INT(10),name VARCHAR(50));
mysql> insert into work_user values(01,'zhangsan'),(02,'lisi');

  • 6.3 登录主从mysq数据库查看新写入的两条数据

  • 6.4 再登录到mysql-proxy，查询数据，可以正常查询
  • 6.5 登录从数据库服务器关闭mysql同步进程：stop slave;，这时再登录mysql-proxy会发现能看到表，但是查询不出数据，会报错：Table '*****' doesn't exist
  • 6.6 到此实现了读写分离的效果

MySQL数据库主从复制

MySQL数据库主从复制

简介

  • 主服务器：开启二进制日志；配置唯一的server-id；获得master二进制日志文件名及位置；创建一个用于slave和master通信的用户账号。
  • 从服务器：启动中继日志；配置唯一的server-id；使用master分配的用户账号读取master二进制日志；启用slave服务。

主数据库配置

  • 在数据库配置文件中增加配置

    vi /etc/my.cnf后在[mysqld]模块中增加如下部分
[mysqld]
server-id =1                  //数据库ID号，为1到2^32–1之间的一个正整数值，且唯一。 
log_bin=master-bin            //启用二进制日志；mysql-bin可以自定义，也可以加上路径
log_bin_index=master-bin.index
binlog-do-db=practice       //需要同步的二进制数据库名；
slave-skip-errors=1062,1053,1064,1146   //跳过错误，继续执行复制；slave-skip-errors有四个可选值，off、all、ddl_exist_errors和错误代码，没有则默认为off
explicit_defaults_for_timestamp=true    //这是为了取消TIMESTAMP的警告

  • 之后重启服务service mysqld restart，进入mysql数据库，创建从数据库有复制权限的用户：replication slave，命令如下：

    grant replication slave on *.* to 'mysql'@'192.168.110.%' identified by '123456';
//该命令中的192.168.110.%通配符，表示0-255的IP都可访问主服务器，正式环境需要配置指定从服务器IP
flush privileges; //刷新权限

运行show global variables like '%log%';    查看二进制日志是否开启
运行show master status;                    查看主数据库的master状态
运行show global variables like '%server%'; 查看主节点的server id

需要注意：确保主服务器的iptables没有阻断3306的访问端口。

从数据库配置

  • 在数据库配置文件中增加配置

    server-id=2                              //指定唯一的server_id
relay_log=slave-relay-bin               //配置中继日志
relay-log-index=slave-relay-bin.index   //中继日志的索引文件

  • 之后重启服务service mysqld restart，进入mysql数据库，连接master主服务器，命令如下：

    change master to master_host='192.168.110.73',master_port=3306,master_user='mysql',master_password='123456',master_log_file='master-bin.000001',master_log_pos=1778;
//master_log_file与master_log_pos分别对应show master status显示的file列与position列

运行start slave;        启动slave数据同步
运行show slave status\G;以竖行显示slave信息，Slave_IO_Running和Slave_SQL_Running都为yes，则表示同步成功。

测试主从复制

  • 1.在主服务器上登陆mysql，在数据库practice下创建新的表，如tb_test表：

    create table tb_test(ID varchar(10) primary key comment '主键ID',MEMO varchar(500) not null comment '信息');
 插入一条数据:
insert into tb_test(ID,MEMO) values('1','one test');

  • 2.在从服务器上登陆mysql，且进入practice数据库

    show tables;查看是否出现了tb_test表，
select * from tb_test;查看表中是否one test数据存在，证明同步数据成功。

清理内存缓存与交换空间

清理内存缓存与交换空间

1. Linux的内存分配方式

  • Linux服务器为提高效率，会提前申请内存，即使这些内存没有被具体应用使用，Linux也会提前申请这些内存，然后利用这些内存做缓存使用，即将刚打开的文件系统存入cache中，这样对应的服务器free值会越来越少，buffers和cached会越来越大，因此给大家表象就是内存越来越少了；其实，Linux服务器在发现内存不足时，会自动清理cached区域，释放内存，然后继续增大cache，free继续减少。因此，下面这些手动降低内存使用率的方法，其实就是图一时之快。

2. 查看内存使用情况

  • free -h 或者 free -m/g

    -h：用辨识度较高的单位显示。
-m/g：用mB或者gB为单位显示。
total：内存总大小。
used：使用了多少内存。
free：除了buff/cache剩余了多少内存。
shared：共享内存。
buff/cache：缓冲、缓存区内存数。
available：真实剩余的可用内存数。
关系：total = used + free + buff/cache
关系：available 包含 free 和 buff/cache 剩余部分，则是真实剩余内存。

3. 清理缓存

  • 3.1 清理缓存准备
    • 在清理缓存前应该先sync一下，因为系统在操作过程中，会将要操作到的文件资料先保存到buffer中去，以免操作过程中因断电等原因遗失数据，所以在操作过程中会把文件资料先缓存处理。
    • /proc是一个虚拟文件系统，我们可以通过对它的读写操作作为与kernel实体间进行通信的一种手段。也就是说可以通过修改/proc中的文件，来对当前kernel的行为做出调整。也就是说我们可以通过调整/proc/sys/vm/drop_caches来释放内存。

  • 3.2 开始清理缓存

    echo 0 > /proc/sys/vm/drop_caches    
#不释放缓存
echo 1 > /proc/sys/vm/drop_caches    
#释放页面缓存，清空最近访问过的文件页面缓存
echo 2 > /proc/sys/vm/drop_caches    
#释放dentries和inodes缓存，清空文件节点缓存和目录项缓存
echo 3 > /proc/sys/vm/drop_caches    
#释放1和2中的提及的所有缓存，清空1和2所有内容的缓存

  • 3.3 常用命令组合

    • 只清除页面缓存

      sync; echo 1 > /proc/sys/vm/drop_caches

    • 清除目录项和inode

      sync; echo 2 > /proc/sys/vm/drop_caches

    • 清除页面缓存、目录项和i节点

      sync; echo 3 > /proc/sys/vm/drop_caches

    • 上述命令的解释：

      • sync命令将刷新文件系统缓冲区。分号（；）分隔的命令依次执行。在执行序列中的下一个命令之前，shell等待每个命令终止。如内核文档中提到的，写入drop_cache将清除缓存而不会终止任何应用程序或服务。echo命令正在做的任务就是写入文件。
      • 如果你必须清除磁盘缓存，在企业和生产环境中第一个命令（“…echo 1>…”）是最安全的，它将仅仅只清除页面缓存。生产环境中不推荐使用上述第三个选项（”…echo 3 >…”），直到你知道你正在做什么。因为它会清除页面缓存，目录项和i节点。

Nginx转发SSH代理-Stream模块

Nginx转发SSH代理-Stream模块

使用案例

  • 公司的测试主机在内网，外网不能直接访问，可以使用nginx做个tcp代理(使用stream模块)做个端口映射达到可以在本地直接通过外网（可能是已绑定的域名）访问内网测试主机的目的，这里主要做ssh映射。

1.确定内网主机的SSHD服务开启

  • sshd为secure shell的简称，可以通过网络在主机中开机shell的服务默认情况下，sshd服务是开机时自启动的。默认配置文件在/etc/ssh/sshd_config

    #查看服务
systemctl status sshd.service
#重启服务
systemctl restart sshd.service

2.网络防火墙

  • (略)

3.Nginx中stream模块的配置

  • 3.1 stream模块简介

    nginx从1.9.0开始，新增加了一个stream模块，用来实现四层协议的转发、代理或者负载均衡等。使用nginx中的该模块做ssh转发可以连接到外网无法访问的内网主机。
  • 3.2 stream模块编译

    stream模块默认没有编译到nginx， 编译nginx时候需要注意将该模块编译进去：./configure –with-stream即可。

  • 3.3 配置ssh内网转发

    stream {    #stream模块与http模块处于同一级；stream模块编译后需要手动添加；这两个模块可以共存。
    upstream remote01{    
        server 192.168.110.101:22;       #内网服务器101，端口为sshd默认的22
    }
    upstream remote02{
        server 192.168.110.102:22;       #内网服务器102，端口为sshd默认的22
    }

    server {
        listen 2201;                    #监听代理主机的2201端口，通过该端口映射到内网的22端口
        proxy_connect_timeout 1h;
        proxy_timeout 1h;
        proxy_pass remote01;            #转向内网服务器101
    }
    server {
        listen 2202;                    ##监听代理主机的2202端口，通过该端口映射到内网的22端口
        proxy_connect_timeout 1h;
        proxy_timeout 1h;
        proxy_pass remote02;            ##转向内网服务器102
    }
}

注意1：server里面不能够使用location这种语句，否则报错；语法虽与http模块类似，但是有区别的。
注意2：如果与代理主机绑定多个二级域名，在server虚拟机中不可以具体指向二级域名，只需要指明需要监听的代理主机的端口即可，比如上述中的server主机如下设置的话，连接会失败。
server {
        listen ns01.testing.com:2201;   #ns01.testing.com为绑定代理主机的二级域名，通过该域名对应访问内网服务器           
        ......
        proxy_pass remote01;            
    }
    server {
        listen ns02.testing.com:2202;  ##ns02.testing.com为绑定代理主机的二级域名，通过该域名对应访问内网服务器                             
        ......
        proxy_pass remote02;   
    }

  • 3.4 使用shell远程工具通过二级域名访问内网主机时，需要注意填写的端口是代理服务器监听的端口，不是内网服务器sshd默认的22端口。

tomcat使用apr模式运行

tomcat使用apr模式运行

1.安装apr
1.1 在/usr/local解压并进入apr-1.6.5后执行：`./configure --prefix=/usr/local/apr`
1.2 make
1.3 make install
2.安装 apr-iconv
2.1 在/usr/local解压并进入apr-iconv-1.2.2后执行：`./configure --prefix=/usr/local/apr-iconv --with-apr=/usr/local/apr`
2.2 make
2.3 make install
3.安装apr-util
3.0需要安装expat库：yum install -y  expat-devel
3.1 在/usr/local解压并进入apr-util-1.6.1后执行：`./configure --prefix=/usr/local/apr-util --with-apr=/usr/local/apr --with-apr-iconv=/usr/local/apr-iconv/bin/apriconv`
3.2 make
3.3 make install
4. 安装tomcat-native（Tomcat自带tomcat-native.tar.gz，在bin下）
4.1在tomcat/bin下解压并进入tomcat-native-1.2.21-src/native后执行：`./configure --with-apr=/usr/local/apr`
4.2 make
4.3 make install
5.配置APR本地库到系统共享库搜索路径中
vi tomcat/bin/catalina.sh 在虚拟机启动参数JAVA_OPTS中添加java.library.path参数，指定apr库的路径：
# Register custom URL handlers
# Do this here so custom URL handles (specifically 'war:...') can be used in the security policy
JAVA_OPTS="$JAVA_OPTS -Djava.protocol.handler.pkgs=org.apache.catalina.webresources"
JAVA_OPTS="$JAVA_OPTS -Djava.library.path=/usr/local/apr/lib"
6.修改protocol
<Connector  executor="tomcatThreadPool"  port="8080" protocol="org.apache.coyote.http11.Http11AprProtocol" ....../>
7.企业apr生命周期监听器（不使用SSL）
<Listener className="org.apache.catalina.core.AprLifecycleListener" SSLEngine="off" />
