磁盘挂载

磁盘挂载

1. 是否能够挂载

  • 如下所示

    fdisk -l          查看硬盘，发现没有 /dev/*vdb 表明云服务无数据盘，无需进行挂载

2. 如何挂载数据盘

  • 如下所示

    a. 执行 fdisk -S 56 /dev/xvdb 命令，对数据盘进行分区
b. 依次输入“n”、“p”、“1”，两次回车，“wq”，分区就开始了
c. 执行 echo '/dev/xvdb1 /mnt ext3 defaults 0 0' >> /etc/fstab （“/mnt”就是要把数据盘挂载到的文件夹）命令写入新分区信息。
d. 使用 cat /etc/fstab 命令查看，出现信息（“defaults 0 0”）就表示写入成功。

3. 其他命令

  • 如下所示

    blkid                     查看已经存在哪些盘
fuser -kvm /dev/vda1    查看使用情况
ls /mnt 
umount /dev/sdb1          解除挂载
mount /dev/sdb1            挂载

配置MongoDB访问账号&密码

配置MongoDB访问账号&密码

1. 为什么？

  • MongoDB默认安装完成后，只允许本地连接，同时不需要使用任何账号密码就可直接连接MongoDB，这样容易被攻击，比如让支付一些比特币，所以为了避免这些不必要的麻烦，需要给Mongo设置一个账号密码

2. 创建管理员用户

  • 步骤如下所示

    cd /usr/local/mongodb/bin/    进入MongoDB的bin目录下
mongod -f mongodb.conf        启动MongoDB服务
/sbin/iptables -I INPUT -p tcp --dport 27017 -j ACCEPT       开放端口
./mongo                                   连接MongoDB服务

>use admin
>db.createUser({user:"admin",pwd:"123456",roles:[{role:"userAdminAnyDatabase",db:"admin"}]})

  • 认证管理员用户

    ps -ef|grep mongodb    sudo kill ****          关闭MongoDB服务 
mongod -f mongodb.conf --auth                  启动MongoDB服务，启用用户认证
./mongo                                        连接MongoDB服务

>use admin
>db.auth("admin", "123456")                    结果返回“1”，表示成功

3. 创建数据库用户（以myinfo数据库为例）

  • 单个数据库赋予最大权限

    >use myinfo
>db.createUser({user:"max",pwd:"123456",roles:[{role:"dbOwner",db:"myinfo"}]})
>db.auth("max", "123456")

  • 单个数据赋予读写权限

    >use myinfo
>db.createUser({user:"min",pwd:"123456",roles:[{role:"readWrite",db:"myinfo"}]})
>db.auth("min", "123456")

4. 删除某个用户（以myinfo数据库为例，删除max用户）

  • 如下所示

    >use muinfo
>db.dropUser("max")        结果返回“true”，表示成功

5. 角色权限说明

  • 如下所示

    Read：                    允许用户读取指定数据库
readWrite：                允许用户读写指定数据库
dbAdmin：                允许用户在指定数据库中执行管理函数，如索引增删，查看统计或访问system.profile
dbOwner：                单数据库最大权限
userAdmin：                允许用户向system.users集合写入，可找指定数据库里增删和管理用户
clusterAdmin：            只在admin数据库中可用，赋予用户所有分片和复制集相关函数的管理权限
readAnyDatabase：        只在admin数据库中可用，赋予用户所有数据库的读权限
readWriteAnyDatabase：    只在admin数据库中可用，赋予用户所有数据库的读写权限
userAdminAnyDatabase：    只在admin数据库中可用，赋予用户所有数据库的userAdmin权限
dbAdminAnyDatabase：        只在admin数据库中可用，赋予用户所有数据库的dbAdmin权限
root：                    只在admin数据库中可用，超级账号，超级权限

疑问

  • 要不要在配置文件中加auth = true呢

mysql日志文件

mysql日志文件

1. 简介

  • 1.1 mysql日志的种类，一般来说，日志有三种，分别为：
    • 错误日志：log_error (记录启动，运行，停止mysql时出现的信息)
    • 二进制日志：log_bin （记录所有更改数据的语句，还用于复制，恢复数据库用）
    • 查询日志：log （记录建立的客户端连接和执行的语句，包括sql查询语句）
    • 慢查询日志: slow_query_log（记录所有执行超过long_query_time秒的所有查询）
  • 1.2 在测试过程中，比较关注的是慢查询日志

2. 日志类型

  • 2.1 错误日志

    配置  在my.cnf中添加：`log-error=/usr/local/mysql/data/`error.log
验证  `mysql> show variables like 'log_error';`

  • 2.2 二进制日志

    配置 在my.cnf中添加： log_bin=logbin
验证 mysql> show variables like 'log_bin';

  • 2.3 查询日志

    配置 在my.cnf中添加：log=/usr/local/mysql/data/query_log.log
验证 查询日志是纯文本格式，可以使用tai直接打开查看。如果是查看当前执行的SQL，可在mysql控制台中执行：mysql>show processlist;
注意 由于查询日志纪录了所有数据库的操作，因此对于访问频繁的应用，该日志会对系统性能造成一定影响，通常建议关闭此日志。

  • 2.4 慢查询日志

    • （参照该笔记中《MySQL慢日志查询》一篇）

Nginx日志文件

Nginx日志文件

1. 简介

  • 1.1 通过访问日志access log，可以得到用户地域来源、跳转来源、使用终端、某个URL访问量等相关信息；通过错误日志error log，可以得到系统某个服务或server的性能瓶颈等
  • 1.2 如果配置日志部分，日志生成到Nginx根目录logs/access.log文件中，默认使用“main”日志格式，也可以自定义格式。

2.日志类型

  • 2.1 access.log

    $remote_addr    客户端的ip地址(代理服务器，显示代理服务ip)
$remote_user    用于记录远程客户端的用户名称（一般为“-”）
$time_local    用于记录访问时间和时区
$request        用于记录请求的url以及请求方法
$status        响应状态码，例如：200成功、404页面找不到等。
$body_bytes_sent    给客户端发送的文件主体内容字节数
$http_referer    可以记录用户是从哪个链接访问过来的
$http_user_agent    用户所使用的代理（一般为浏览器）
$http_x_forwarded_for    可以记录客户端IP，通过代理服务器来记录客户端的ip地址

Tomcat日志文件

Tomcat日志文件

1. 简介

  • 1.1 Tomcat 默认的日志库对JDK自带的java.util.logging API 进行了重新实现，这些实现被称为 “JULI” ，里面包含了一个自定义的LogManager类，它能够区别出运行在 Tomcat 容器中的多个不同的Web应用以及它们的类加载器，从而可以支持不同的应用使用各自独立的日志配置。
  • 1.2 可以从 Tomcat 全局和 Web 应用两个层面对 Tomcat 默认的JULI进行日志配置：
    • 全局配置通常使用 tomcat下/conf/logging.properties文件进行配置
    • 应用配置则是使用 WEB-INF/classes/logging.properties文件进行配置
  • 1.3 处理器默认的日志级别是 INFO，可选的日志级别从高到低依次为：SEVERE > WARNING > INFO > CONFIG > FINE > FINER > FINEST > ALL > OFF。

2. 日志类型

  • 2.1 catalina引擎的日志文件，文件名：catalina.日期.log，如 catalina.2019-04-26.log
    • 主要是记录tomcat启动时候的信息，类似于我们在eclipse中启动服务的时候在控制台看到的信息。在里面可以看到启动的JVM参数以及操作系统等日志信息。如果没有使用log4j等日志机制，系统的的错误信息与打印语句也在这个日志文件中记录
  • 2.2 Tomcat下内部代码丢出的日志，文件名：localhost.日期.log，如 localhost.2019-04-26.log
    • jsp页面内部错误的异常，org.apache.jasper.runtime.HttpJspBase.service类丢出的，日志信息就在该文件
  • 2.3 Tomcat下默认manager项目专有的日志，文件名：manager.日期.log，如 manager.2019-04-26.log
  • 2.4 tomcat下默认manager项目的日志信息，文件名：host-manager.2019-04-26.log
  • 2.5 控制台输出的日志，Linux下默认重定向到catalina.out
  • 2.6 存放访问tomcat的请求信息，如 localhost_access_log.2019-04-26.txt
    • 配置文件在 tomcat 的安装目录/conf/server.xml中

定时任务crontab命令

定时任务crontab命令

1. 简介

  • 1.1 linux 系统是由 cron (crond) 这个系统服务来控制的。Linux 系统上面原本就有非常多的默认启动的计划性工作，由于使用者自己也可以设置计划任务，所以， Linux 系统也提供了使用者控制计划任务的命令 :crontab 命令
  • 1.2 当安装完成操作系统后，默认会安装此服务工具，并且会自动启动crond进程，crond进程会周期性检查是否有要执行的任务，如果有要执行的任务，则自动执行该任务。

2. 验证

  • 2.1 检查是否安装了crontab

    shell命令行：rpm -qa | grep crontab
目前版本 crontabs-1.11-6.20121102git.el7.noarch

  • 2.2 如果提示未安装请自行安装，yum install crontabs

  • 2.3 服务操作说明

    /sbin/service crond start         //启动服务
/sbin/service crond stop         //关闭服务
/sbin/service crond restart     //重启服务
/sbin/service crond reload         //重新载入配置
/sbin/service crond status         //启动服务

chkconfig –level 35 crond on       //加入开机自动启动

3. 系统任务调度

  • 3.1 系统周期性所要执行的工作，比如写缓存数据到硬盘、日志清理等。在/etc目录下有一个crontab文件，这个就是系统任务调度的配置文件。

  • 3.2 文件说明

    /etc/cron.deny      该文件中所列用户不允许使用crontab命令
/etc/cron.allow     该文件中所列用户允许使用crontab命令
/var/spool/cron/    所有用户crontab文件存放的目录,以用户名命名

4. 用户任务调度

  • 4.1 用户定期要执行的工作，比如用户数据备份、定时邮件提醒等。所有用户定义的crontab 文件都被保存在 /var/spool/cron目录中，其文件名与用户名一致。

  • 4.2 文件说明

    • 用户所建立的crontab文件中，每一行都代表一项任务，每行的每个字段代表一项设置，它的格式共分为六个字段，前五段是时间设定段，第六段是要执行的命令段

    • 格式如下：

      minute hour day month week command
其中：
minute：          表示分钟，可以是从0到59之间的任何整数。
hour：             表示小时，可以是从0到23之间的任何整数。
day：               表示日期，可以是从1到31之间的任何整数。
month：          表示月份，可以是从1到12之间的任何整数。
week：            表示星期几，可以是从0到7之间的任何整数，这里的0或7代表星期日。
command：        要执行的命令，可以是系统命令，也可以是自己编写的脚本文件。

    • 在以上各个字段中，还可以使用以下特殊字符：

      星号（*）：    代表所有可能的值，例如month字段如果是星号，则表示在满足其它字段的制约条件后每月都执行该命令操作。
逗号（,）：    可以用逗号隔开的值指定一个列表范围，例如，“1,2,5,7,8,9”
中杠（-）：    可以用整数之间的中杠表示一个整数范围，例如“2-6”表示“2,3,4,5,6”
正斜线（/）：    可以用正斜线指定时间的间隔频率，例如“0-23/2”表示每两小时执行一次。同时正斜线可以和星号一起使用，例如*/10，如果用在minute字段，表示每十分钟执行一次。

  • 4.3 环境变量EDITOR

    • 在考虑向cron进程提交一个crontab文件之前，首先要设置环境变量EDITOR。cron进程根据它来确定使用哪个编辑器编辑 crontab文件。多数LINUX用户都使用vi，那么需要编辑$HOME目录下的.bash_profile文件。

      cd /root
vi .bash_profile后添加如下：
EDITOR=vi
export EDITOR

5. crontab命令详解

  • 5.1 命令参数

    crontab -e              新建、编辑crontab文件
crontab -l               列出crontab文件
crontab -r               列出crontab文件

  • 5.2 实际应用

    • 5.2.1 由于实际工作中，在crontab中的命令均为shell脚本，而crontab本身没有环境给该shell运行，所以需要在shell脚本中最顶头添加如下命令：

      source /etc/profile
. ~/.bash_profile
#!/bin/bash
......

    • 5.2.2 备份数据库的实例（每天凌晨1:30进行备份）

      shell命令行执行：crontab -e 
添加如下命令：
30 1 * * *  /root/backup_database.sh

    • 5.2.3 backup_database.sh脚本如下所示：

      source /etc/profile        
. ~/.bash_profile        
#!/bin/bash

mysql_host="192.168.110.**"    
mysql_user="root"    
mysql_pwd="123456"    
backup_database="test_oa"
backup_date=$(date +%Y%m%d_%H%M%S)

mysqldump -h$mysql_host -u$mysql_user -p$mysql_pwd $backup_database > /root/test_oa_$backup_date.sql

MySQL慢日志查询

MySQL慢日志查询

1. 简介

  • 1.1 MySQL的慢查询日志是MySQL提供的一种日志记录，用来记录在MySQL中响应时间超过long_query_time值的语句。long_query_time的默认值为10，意思是运行10S以上的语句。
  • 1.2 默认情况下，MySQL数据库并不启动慢查询日志，需要手动来设置这个参数。如果不是调优需要的话，一般不建议启动该参数，因为开启慢查询日志会或多或少带来一定的性能影响。
  • 1.3 慢查询日志支持将日志记录写入文件，也支持将日志记录写入数据库表。

2. 查询

  • slow_query_log

    mysql> show variables like '%slow_query_log%';
注：
slow_query_log   是否开启慢查询日志，1表示开启，0表示关闭；默认为OFF
slow-query-log-file    MySQL数据库慢查询日志存储路径。可不设置该参数，系统默认host_name-slow.log

  • long_query_time

    mysql> show variables like 'long_query_time';
注：
long_query_time    慢查询阈值，当查询时间多于设定的阈值时，开始记录日志。默认值为10秒

  • log_output

    mysql> show variables like '%log_output%';
注：
log_output 指定日志的存储方式。'FILE'表示将日志存入文件，默认值是'FILE'。'TABLE'表示将日志存入数据库的mysql.slow_log表中

  • log_queries_not_using_indexes

    mysql> show variables like 'log_queries_not_using_indexes';
注：
log_queries_not_using_indexes   1表示开启，0表示关闭；默认为OFF。未使用索引的查询也被记录到慢查询日志中；如果调优的话，建议开启这个选项。另外，开启了这个参数，其实使用full index scan的sql也会被记录到慢查询日志

  • log_slow_admin_statements

    mysql> show variables like 'log_slow_admin_statements';
注：
log_slow_admin_statements   1表示开启，0表示关闭；默认为OFF。是否将慢管理语句例如ANALYZE TABLE和ALTER TABLE等记入慢查询日志

  • slow_queries

    mysql> show global status like '%slow_queries%';
注：
查询有多少条慢查询记录，可以使用系统变量slow_queries

3. 配置

  • 修改my.cnf文件，增加或修改参数后，然后重启MySQL服务器，如下所示:

    slow_query_log=1
slow_query_log_file=/usr/local/mysql/data/localhost-slow.log
long_query_time=5
log_output='FILE'    #可以不设置，出于性能考虑，默认即可
log_queries_not_using_indexes=1   可选项

4. 日志分析工具mysqldumpslow

  • 4.1 在实际生产环境中，如果要手工分析日志，查找、分析SQL，显然是个体力活，MySQL提供了日志分析工具mysqldumpslow

  • 4.2 常用参数

    -s, 是表示按照何种方式排序，有下面7种方式
    c: 访问计数
    l: 锁定时间
    r: 返回记录
    t: 查询时间
    al:平均锁定时间
    ar:平均返回记录数
    at:平均查询时间
-t, 是top n的意思，即为返回前面多少条的数据；
-g, 后边可以写一个正则匹配模式，大小写不敏感的

  • 4.3 常用命令

    得到返回记录集最多的10个SQL。
mysqldumpslow -s r -t 10 /database/mysql/mysql01_slow.log

得到访问次数最多的10个SQL
mysqldumpslow -s c -t 10 /database/mysql/mysql01_slow.log

得到按照时间排序的前10条里面含有左连接的查询语句。
mysqldumpslow -s t -t 10 -g “left join” /database/mysql/mysql06_slow.log

另外建议在使用这些命令时结合 | 和more 使用 ，否则有可能出现刷屏的情况。
mysqldumpslow -s r -t 20 /mysql/mysql06-slow.log | more

MySQL监控与调优--初级

MySQL监控与调优–初级

1. 简介

  • 1.1 为了保证数据库的平稳运行，可以主动监控以下三个与性能及资源利用率相关的指标：连接情况、缓冲池使用情况、查询执行性能
  • 1.2 本篇笔记讨论的一些监控策略只适用于 MySQL 5.6 与 5.7 版本

2. 连接情况

  • 2.1 监控客户端连接情况相当重要，因为一旦可用连接耗尽，新的客户端连接就会遭到拒绝。MySQL 默认的连接数限制为 151，可通过下面的查询加以验证：

    mysql命令行：SHOW VARIABLES LIKE 'max_connections';

  • 2.2 常规情况下，Linux应该能够支持 500 到 1000 个同时连接。如果可用的 RAM 较大，且每个连接的工作量较低或目标响应时间较为宽松，则最多可处理 10000 个连接。而 Windows 能处理的连接数一般不超过 2048 个，这是由于该平台上使用的 Posix 兼容层。

  • 2.3 修改连接数

    第一种方式：（可在系统运行时进行调整，服务器重启时恢复为默认值）
mysql命令行：SET GLOBAL max_connections = 200;

第二种方式：（可在my.cnf配置文件中添加配置，永久地改变连接数限制）
max_connections = 200

  • 2.4 监控连接使用率

    查看当前打开的连接数
show status like '%Threads_connected%';

  • 2.5 同一个ip在短时间内产生太多（超过mysql数据库max_connection_errors的最大值）中断的数据库连接而导致的阻塞，可以设置该max_connection_errors的数值

    max_connect_errors=1000

  • 2.6 MySQL会根据配置文件会限制server接受的数据包的大小。如果写入大数据时，因为默认的配置太小，插入和更新操作会因为 max_allowed_packet 参数限制，而导致失败

    max_allowed_packet=100M

3. 缓冲池使用情况

  • 3.1 存储引擎层：InnoDB_Buffer_Pool

    • 3.1.1 MySQL 默认的存储引擎 InnoDB 使用了一片称为缓冲池的内存区域，用于缓存数据表与索引的数据。缓冲池指标属于资源指标，而非工作指标。如果数据库性能开始下滑，而磁盘 I/O 在不断攀升，扩大缓冲池往往能带来性能回升。
    • 3.1.2 默认设置下，缓冲池的大小通常相对较小，为 128MiB。不过，MySQL 建议可将其扩大至专用数据库服务器物理内存的 80% 大小。

    • 3.1.3 缓冲池大小调整操作是分块进行的，缓冲池的大小必须为块的大小乘以实例的数目再乘以某个倍数。

      innodb_buffer_pool_size = N * innodb_buffer_pool_chunk_size * innodb_buffer_pool_instances
以上两个参数的值都可以通过如下方式进行检查：
SHOW GLOBAL VARIABLES LIKE "innodb_buffer_pool_chunk_size";
SHOW GLOBAL VARIABLES LIKE "innodb_buffer_pool_instances";
如果 innodb_buffer_pool_chunk_size 查询没有返回结果，则表示在你使用的 MySQL 版本中此参数无法更改，其值为 128 MiB。

    • 3.1.4 查询验证

      查看 InnoDB_Buffer_Pool 大小：
SHOW VARIABLES LIKE '%innoDB_buffer_pool%';
查询 InnoDB_buffer_pool 当前使用情况：
SHOW STATUS LIKE '%InnoDB_buffer_pool %';

    • 3.1.5 调优参考计算方法

      show global status like 'Innodb_buffer_pool_pages_data';
show global status like 'Innodb_buffer_pool_pages_total';

val = Innodb_buffer_pool_pages_data / Innodb_buffer_pool_pages_total * 100%

val > 95% 则考虑增大 innodb_buffer_pool_size， 建议使用物理内存的75%

val < 95% 则考虑减小 innodb_buffer_pool_size， 建议设置为：Innodb_buffer_pool_pages_data * Innodb_page_size * 1.05 / (1024*1024*1024)

注：show global status like 'Innodb_page_size';    默认值为16K

    • 3.1.6 在my.cnf中设置

      innodb_buffer_pool_size = 2147483648         #设置2G
innodb_buffer_pool_size = 500M               #设置500M

    • 3.1.7 设置要根据自己的实际情况来设置，如果设置的值不在合理的范围内，并不是设置越大越好，可能设置的数值太大体现不出优化效果，反而造成系统的swap空间被占用，导致操作系统变慢，降低sql查询性能。

  • 3.2 mysql层：查询缓存 Query Cache

    • 3.2.1 该缓存存储select的完整结果，当有同样的查询，则跳过解析，优化和执行的阶段，直接返回结果，当这条缓存select语句相关的表有变化时，此缓存失效。开启可以提高性能，但是对于读写增加了开销，读的时候要先检查缓存，对于写，写入之后要更新缓存。但是一般这种开销比较小，但是也要根据业务特征权衡。

    • 3.2.2 在my.cnf中设置

      query_cache_type=on        on 开启      off 关闭
query_cache_size=200M      一般建议不超过256M
query_cache_limit=1M       限制储存结果的大小，如果查出来的结果比这个大，就不会被缓存

    • 3.2.3 查询验证

      查询quache大小：
SHOW VARIABLES LIKE '%query_cache%';
查询quache当前使用情况：
SHOW STATUS LIKE '%Qcache%';

4. 查询执行性能

  • 4.1 QPS:每秒的查询数

    QPS计算方法
Questions = SHOW GLOBAL STATUS LIKE 'Questions';
Uptime = SHOW GLOBAL STATUS LIKE 'Uptime';

QPS=Questions/Uptime

  • 4.2 TPS:每秒的事物量（commit与rollback的之和）

    TPS计算方法
Com_commit = SHOW GLOBAL STATUS LIKE 'Com_commit';
Com_rollback = SHOW GLOBAL STATUS LIKE 'Com_rollback';
Uptime = SHOW GLOBAL STATUS LIKE 'Uptime';

TPS=(Com_commit + Com_rollback)/Uptime

完全备份与还原---mysqldump备份

完全备份与还原—mysqldump备份

1. 简介

  • 这篇笔记主要描述记录如何使用mysqldump命令来产生导出文件，和如何导入这些导出文件。

2. 备份数据

  • 2.1 备份一个数据库

    语法：shell模式下，mysqldump -u [username] -p [dbname] > backupName.sql

例如：（使用root用户备份名为test_oa的数据库）
mysqldump -u root -p test_oa > /root/bat_test_oa.sql
其中：
dbname参数表示需要备份的数据库的名称；
backupName.sql参数表示备份文件的名称，文件名前面可以加上一个绝对路径。

  • 2.2 备份数据库中的表

    语法：shell模式下，mysqldump -u [username] -p [dbname] [table1] [table2] ... > backupName.sql

例如：（使用root用户备份名为test_oa的数据库中的userinfo、workinfo两张表）
mysqldump -u root -p test_oa userinfo workinfo> /root/bat_test_oa_uw.sql
其中：table1和table2参数表示需要备份的表的名称，为空则整个数据库备份

  • 2.3 备份多个数据库

    语法：shell模式下，mysqldump -u [username] -p --databases [dbname1] [dbname2] > backupName.sql

其中：dbname1和dbname2参数表示需要备份的数据库的名称

  • 2.4 备份所有数据库

    语法：shell模式下，mysqldump -u [username] -p --all-databases > backupAll.sql

3. 还原数据

  • 3.1 还原一个数据库

    • 当使用上面的mysqldump语句导出文件的时候未使用到–databases或者–all-databases选项时，需要手动创建数据库

      语法：shell模式下，mysqladmin -u [username] -p create [dbname];
                 mysql -u [username] -p [dbname] < backupName.sql

例如：mysqladmin -uroot -p create test_oa;
     mysql -uroot -p test_oa < /root/bat_test_oa.sql
     其中：在导入文件时需要指定数据库名字

  • 3.2 还原多个数据库

    • 当使用上面的mysqldump语句导出文件的时候使用到了–databases或者–all-databases选项后，文件中已经含有了CREATE DATABASE和USE语句，所以不需要再指定一个数据库，可以直接读入所备份文件

      语法：shell模式下，mysql -u [username] -p < backupName.sql
